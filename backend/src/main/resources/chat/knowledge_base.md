# LeadBoard — Полная база знаний AI-ассистента

## О СИСТЕМЕ

LeadBoard — инструмент управления командной разработкой. Синхронизируется с Jira, показывает прогресс по эпикам и стори, прогнозирует сроки завершения, отслеживает метрики команды и качество данных. Поддерживает мультитенантность — каждая компания работает в изолированном пространстве.

### Иерархия задач
```
PROJECT (проект) → EPIC (эпик) → STORY / BUG (стори / баг) → SUBTASK (подзадача)
```
- Subtask — единственное место для оценок времени и логирования
- Bug обрабатывается как Story, но с отдельными SLA-правилами
- Подзадача не существует без стори, стори без эпика

### Роли пользователей
| Роль | Возможности |
|------|-------------|
| ADMIN | Полный доступ: настройки, роли, синхронизация, все команды |
| PROJECT_MANAGER | Управление проектами, RICE-скоринг, кросс-командный обзор |
| TEAM_LEAD | Управление своей командой, приоритеты, участники |
| MEMBER | Просмотр доски, участие в Planning Poker |
| VIEWER | Только просмотр |

## ЭКРАН: BOARD (Доска) — `/`

Главный экран. Иерархическая таблица: эпики (строки) → стори → подзадачи.

### Фильтры (верхняя панель)
- **Поиск по ключу** — фильтрует эпики по подстроке (регистронезависимо)
- **Команда** — мультиселект, показывает цветные точки команд. При выборе одной команды включается drag & drop
- **Статус** — мультиселект по статусу эпика
- **Кнопка «Обновить»** — запускает синхронизацию с Jira
- **«Последняя синхронизация»** — дата/время последнего синка

### Колонки таблицы

**NAME (Название)**
- Иконка типа задачи (из Jira), ключ-ссылка на Jira, название
- 🚩 — эпик приостановлен (flagged)
- Бэдж проекта (если эпик принадлежит проекту)
- Drag-handle (⋮⋮) — только при выборе одной команды

**TEAM (Команда)**
- Цветной бэдж TeamBadge с названием команды

**PRIORITY (Приоритет / AutoScore)**
- Числовой AutoScore, цвет: зелёный >80, жёлтый 40-80, серый <40, красный <0
- Индикатор рекомендации: ↑ поднять / ↓ опустить / ● на месте
- Иконки: 🐞 баг, 🔒 заблокировано, 🚩 приостановлено, ⚡ высокий приоритет
- **При наведении**: тултип с разбивкой по факторам AutoScore

**EXPECTED DONE (Прогноз завершения)**
- Дата прогноза (формат «15 мар»)
- Цветная точка уверенности: 🟢 HIGH / 🟡 MEDIUM / 🔴 LOW
- Дельта от дедлайна: зелёный «on time», оранжевый «+3d», синий «-2d»
- **При наведении**: расписание фаз (SA/DEV/QA), оставшаяся работа, дедлайн
- Для стори: дата + имя исполнителя + расписание фаз

**PROGRESS (Прогресс)**
- Процент (0-100%), полоска прогресса
- Оставшиеся дни (например «5.2d»)
- Разбивка: «2.3d → 8.5d» (залогировано → оценка)
- Формула: progress = (loggedSeconds / estimateSeconds) × 100

**ROLE-BASED PROGRESS (Прогресс по ролям)**
- Для каждой роли (SA/DEV/QA): прогресс-бар + проценты
- Для эпиков в TODO: редактируемые rough estimates (дни по ролям, кликабельные чипы)
- Цвета ролей из WorkflowConfig

**STATUS (Статус)**
- StatusBadge — цветной бэдж, цвет из конфигурации статусов

**ALERTS (Проблемы)**
- Количество нарушений качества данных
- Цвет по серьёзности: 🔴 ERROR / 🟡 WARNING / 🔵 INFO
- **При наведении**: список нарушений с описаниями

### Действия
- **Drag & drop** — перетаскивание эпиков/стори для изменения порядка (при выборе одной команды)
- **Раскрытие** — клик на стрелку показывает дочерние стори/подзадачи
- **Rough estimates** — редактирование оценок для эпиков в статусе TODO
- **Синхронизация** — кнопка «Обновить» запускает импорт из Jira

## ЭКРАН: TIMELINE (Таймлайн) — `/timeline`

Гибридная диаграмма Ганта: прошлое (сплошные бары) + будущее (полосатые бары).

### Элементы управления
- **Команда** — выбор команды
- **Масштаб** — День (40px) / Неделя (120px) / Месяц (100px)
- **Дата снимка** — «Сегодня (live)» или историческая дата из forecast snapshots
- **Легенда** — роли с цветами, факт/прогноз, сегодня, дедлайн

### Левая панель — метки эпиков
Для каждого эпика:
- Иконка типа + ключ-ссылка на Jira
- Индикатор дедлайна: 🟢 вовремя / 🟡 риск (±5 дней) / 🔴 опоздание
- 🚩 если flagged
- StatusBadge + прогресс-бар + процент
- Название (обрезается)
- **При наведении**: тултип с деталями: прогресс, даты, работа по ролям, AutoScore

### Диаграмма Ганта (правая панель)
- **Заголовки**: месяцы/кварталы → недели (только для дней) → дни/недели/месяцы
- **Выходные** — серый фон (только в дневном масштабе)
- **Линия «Сегодня»** — синяя вертикальная линия
- **Линия дедлайна** — красная вертикальная линия (для каждого эпика с due date)

### Бары стори
- Горизонтальные бары, позиционированные по датам начала/конца
- Внутри — цветные сегменты фаз (SA/DEV/QA)
- **Сплошные** — ретроспектива (фактические данные из status_changelog)
- **Полосатые** — прогноз (будущая работа из планирования)
- **Гибридные** — пересекают «сегодня»: левая часть сплошная, правая полосатая
- Красная рамка = заблокировано, оранжевая = flagged
- **При наведении**: ключ, статус, прогресс, расписание фаз с исполнителями, блокировки

### Rough estimate бары
- Пунктирная рамка, штриховка, метка «~SA:5/DEV:3/QA:2д»

### Историческая дата
- Оранжевый баннер «Просмотр снэпшота от {дата}»
- Прогноз из сохранённого снимка, ретроспектива всегда live

## ЭКРАН: METRICS (Метрики команды) — `/metrics`

Дашборд производительности команды за выбранный период.

### Элементы управления
- Выбор команды, период (7/14/30/60/90 дней), тип задачи

### Карточки метрик

**DSR Actual (Gauge)**
- Формула: DSR = (дни_в_работе − дни_паузы) / оценка_в_днях
- дни_в_работе: сумма рабочих дней в статусах IN_PROGRESS (из status_changelog)
- Периоды: вход в IN_PROGRESS → выход из IN_PROGRESS. Несколько периодов суммируются
- Открытый период (ещё в работе) → конец = сегодня
- Fallback: если нет changelog → startedAt → doneAt (историческая совместимость)
- дни_паузы: рабочие дни под флагом Jira, только внутри периодов IN_PROGRESS
- оценка: сумма original_estimate подзадач / 8ч. Fallback: rough estimates эпика
- Шкала: ≤1.1 зелёный (вовремя), 1.1-1.5 жёлтый, >1.5 красный

**DSR Forecast (Gauge)**
- Формула: DSR_Forecast = рабочие_дни / прогнозные_дни
- прогнозные_дни: из forecast snapshot (planned_start → planned_end)

**Throughput (карточка)**
- Количество завершённых задач за период (эпики + стори + подзадачи)

**On-Time Rate (карточка)**
- Процент эпиков с DSR ≤ 1.1

### Role Load (Загрузка по ролям)
- Горизонтальные бары: назначенные часы / ёмкость для каждой роли
- Статусы: OVERLOAD (>100%), NORMAL (50-100%), IDLE (<50%), NO_CAPACITY (0)

### DSR по эпикам (таблица)
Колонки: Эпик, Статус, Оценка(д), В работе(д), Пауза(д), Эффект.(д), DSR

### Forecast Accuracy (Точность прогноза)
- accuracy_ratio = planned_days / actual_days
- schedule_variance = actual_end − planned_end (в днях)
- Статусы: ON_TIME / EARLY / LATE

### Throughput Chart (График)
- Столбчатая диаграмма по неделям (эпики/стори/подзадачи)
- Красная линия — 4-недельная скользящая средняя

### Time in Status (Время в статусах)
- Двухосевой график: количество переходов (бары) + медиана/P85/P99 часов (линии)

### Assignee Table (Таблица участников)
Колонки: Имя, Закрыто задач, Avg Lead Time, Avg Cycle Time, DSR, Velocity, Тренд

### Velocity Chart (Скорость)
- Ёмкость vs залогированные часы по неделям
- Утилизация: (залогировано / ёмкость) × 100%

### Epic Burndown (Сгорание)
- Ideal line (линейный спуск) vs Actual line (реальное закрытие стори)

## ЭКРАН: DATA QUALITY (Качество данных) — `/data-quality`

Отчёт о проблемах с данными в Jira.

### Фильтры
- Команда, серьёзность (ERROR/WARNING/INFO), правило

### Карточки
- Всего задач, Ошибки (красный), Предупреждения (жёлтый), Информация (серый)

### Таблица нарушений
- Ключ-ссылка, тип, название, статус, серьёзность (макс.), количество проблем
- Раскрываемая строка — детали каждого нарушения

### Все правила качества данных

**ERROR (блокируют планирование):**
- EPIC_NO_TEAM — эпик без команды
- EPIC_OVERDUE — дедлайн в прошлом, не завершён
- TIME_LOGGED_NOT_IN_SUBTASK — время залогировано не на подзадаче
- CHILD_IN_PROGRESS_EPIC_NOT — ребёнок в работе, эпик нет
- SUBTASK_IN_PROGRESS_STORY_NOT — подзадача в работе, стори нет
- SUBTASK_WORK_NO_ESTIMATE — время залогировано без оценки
- EPIC_DONE_OPEN_CHILDREN — эпик закрыт, есть открытые стори
- STORY_DONE_OPEN_CHILDREN — стори закрыта, есть открытые подзадачи
- STORY_BLOCKED_BY_MISSING — заблокировано несуществующей задачей
- STORY_CIRCULAR_DEPENDENCY — циклическая зависимость
- BUG_SLA_BREACH — баг превысил SLA
- SUBTASK_TIME_LOGGED_BUT_TODO — время залогировано, но статус TODO

**WARNING:**
- TIME_LOGGED_WRONG_EPIC_STATUS — логирование при неверном статусе эпика
- EPIC_NO_ESTIMATE — нет оценки (ни rough, ни подзадач)
- EPIC_TEAM_NO_MEMBERS — команда без активных участников
- SUBTASK_NO_ESTIMATE — подзадача без оценки
- SUBTASK_OVERRUN — превышение оценки на 50%+ (порог 1.5×)
- EPIC_FORECAST_LATE — прогноз позже дедлайна
- EPIC_IN_PROGRESS_NO_STORIES — эпик в работе без стори
- STORY_IN_PROGRESS_NO_SUBTASKS — стори в работе без подзадач
- STORY_NO_SUBTASK_ESTIMATES — нет оценок подзадач (нельзя планировать)
- STORY_BLOCKED_NO_PROGRESS — заблокировано >30 дней без прогресса по блокеру
- RICE_MISSING_ASSESSMENT — нет RICE-оценки для Planned+ задачи
- BUG_STALE — баг без обновлений >14 дней
- STORY_FULLY_LOGGED_NOT_DONE — всё время списано, но не закрыта
- SUBTASK_DONE_NO_TIME_LOGGED — подзадача закрыта без логирования
- SUBTASK_TIME_LOGGED_WHILE_EPIC_FLAGGED — логирование при паузе эпика

**INFO:**
- EPIC_NO_DUE_DATE — нет дедлайна

## ЭКРАН: BUGS (Метрики багов) — `/bug-metrics`

### Карточки
- **Open Bugs** — количество открытых
- **SLA Compliance** — % решённых в рамках SLA (зелёный ≥90%, жёлтый ≥70%, красный <70%)
- **Avg Resolution** — среднее время решения (формат «5д 3ч»)
- **Stale Bugs** — без обновлений >14 дней

### Priority Breakdown (таблица)
Колонки: Приоритет, Open, Resolved, Avg Resolution, SLA Limit, Compliance

### Open Bugs (таблица)
Колонки: Ключ, Название, Приоритет, Статус, Возраст (дни), SLA (Breach/OK)

### SLA по приоритетам (по умолчанию)
- Highest: 24ч, High: 72ч (3д), Medium: 168ч (7д), Low: 336ч (14д), Lowest: 672ч (28д)
- Настраивается в Settings → Bug SLA
- Breach = время_решения > порог_SLA
- Stale = дней_без_обновлений > 14

## ЭКРАН: PROJECTS (Проекты) — `/projects`

### Карточка проекта (свёрнутая)
- Иконка + ключ-ссылка + название + менеджер (аватар)
- RICE Score Badge (зелёный ≥70, оранжевый ≥40, серый <40)
- Статус, прогресс-бар, оставшиеся эпики, Expected Done

### Прогресс проекта
- По времени: (totalLogged / totalEstimate) × 100%
- Fallback по эпикам: (completedEpics / totalEpics) × 100%
- Expected Done = MAX(endDate) незавершённых эпиков

### Карточка проекта (развёрнутая)
- Описание проекта
- Таблица эпиков: ключ, статус, команда (TeamBadge), прогресс, дата, alignment
- Рекомендации (жёлтый блок): EPIC_LAGGING, EPIC_NO_FORECAST, RICE_NOT_FILLED
- Alignment Badge: ✓ (вовремя, ≤2д) / ⚠️+Xd (отстаёт >2д от среднего)

## МЕТРИКИ И ФОРМУЛЫ

### DSR (Delivery Speed Ratio)
```
DSR = (дни_в_работе − дни_паузы) / оценка_в_днях
```

**Дни в работе (status-based)**:
- Считаются из status_changelog: суммируются все периоды нахождения в статусах IN_PROGRESS
- Вход: переход в IN_PROGRESS статус. Выход: переход из IN_PROGRESS в другой статус
- Несколько периодов суммируются (epic может несколько раз входить/выходить из работы)
- Открытый период (эпик ещё в работе) → конец = сегодня
- Fallback для эпиков без changelog: startedAt → doneAt (или сегодня)
- Рабочие дни по календарю РФ (без выходных/праздников, WorkCalendarService)

**Дни паузы (flagged)**:
- Рабочие дни под флагом Jira, считаются **только внутри периодов IN_PROGRESS**
- Источник: flag_changelog (история флагов)

**Эффективные дни**:
- `effective = дни_в_работе − дни_паузы` (минимум 1)

**Оценка в днях (estimate)**:
1. Сумма `original_estimate_seconds` всех подзадач (Epic→Story→Subtask), делённая на 28800 (8ч × 3600с)
2. Fallback: сумма rough estimates эпика по всем ролям в днях
3. Если нет оценок → DSR = null (не считается)

**Интерпретация**:
- DSR = 1.0 — точно по оценке
- DSR < 1.0 — быстрее оценки
- DSR > 1.0 — медленнее оценки
- ≤1.1 вовремя (зелёный), 1.1-1.5 приемлемо (жёлтый), >1.5 перерасход (красный)

**Пример**: Эпик LB-95 был в IN_PROGRESS с 23.01 по 04.03 (29 рабочих дней), пауза = 0, эффект. = 29. Оценка подзадач = 80ч = 10д. DSR = 29/10 = 2.90 (красный — перерасход)

### AutoScore (9 факторов)
| Фактор | Макс | Описание |
|--------|------|----------|
| Status | 30 | Acceptance=30, Developing=25, Planned=15, New=0 |
| Progress | 10 | (logged / estimate) × 10 |
| Due Date | 25 | Экспоненциальный рост к дедлайну |
| Priority | 20 | Highest=20, High=15, Medium=10, Low=5 |
| Size | 5 | Инверсия: меньше = выше, нет оценки = −5 |
| Age | 5 | log(дни_от_создания) |
| RICE Boost | 15 | (normalizedScore / 100) × 15 |
| Alignment | 10 | дни_отставания × 1, макс 10 |
| Flagged | −100 | Если работа приостановлена |

### RICE Score
```
RICE = (Reach × Impact × Confidence) / Effort
Normalized = (raw − min) / (max − min) × 100 × strategic_weight
```
- Reach: охват пользователей (0-100)
- Impact: ценность (0-100)
- Confidence: уверенность (0.4-1.0)
- Effort: трудозатраты в человеко-месяцах (авто из подзадач или T-shirt)

### Velocity
```
Utilization = (logged_hours / capacity_hours) × 100%
```
- Зелёный 85-110%, жёлтый 70-130%, красный остальное

### Lead Time vs Cycle Time
- Lead Time = дата_завершения − дата_создания (всё время)
- Cycle Time = дата_завершения − дата_начала_работы (без ожидания)

### Грейды и ёмкость
- Senior (0.8): 8ч/день ÷ 0.8 = 10 эффективных часов
- Middle (1.0): 8ч/день = 8 эффективных часов
- Junior (1.5): 8ч/день ÷ 1.5 = 5.3 эффективных часа

### Risk Buffer
```
adjusted_estimate = estimate × (1 + 0.20) = estimate × 1.2
```

### Forecast Accuracy
```
accuracy_ratio = planned_days / actual_days
schedule_variance = actual_end − planned_end
```
- ON_TIME: variance ≤ 0, LATE: >0, EARLY: <0

### Throughput
- Количество завершённых задач за период (группируется по неделям)
- Скользящая средняя: sum(4 недели) / 4

### Time in Status
- Среднее/медиана/P85/P99 времени в каждом статусе
- Источник: status_changelog (история переходов)

### Bus Factor
- Количество экспертов (level ≥4) по компоненту
- CRITICAL: 0 экспертов, WARNING: 1, OK: 2+

## АЛГОРИТМ ПЛАНИРОВАНИЯ (Unified Planning)

### Принцип: «Заканчивай начатое»
Эпики планируются в порядке ручной сортировки (manual_order). Pipeline строгий: SA → DEV → QA.

### Шаги
1. Загрузить конфиг (risk_buffer, коэффициенты, WIP-лимиты)
2. Построить расписания участников (AssigneeSchedule)
3. Заблокировать даты отсутствий
4. Для каждого эпика (по порядку):
   - Загрузить стори, отсортировать по зависимостям (топологическая сортировка)
   - Для каждой стори:
     - Извлечь часы по ролям из подзадач
     - Применить risk buffer (×1.2)
     - Для каждой роли в pipeline:
       - Найти лучшего исполнителя (по дате завершения + компетенции)
       - Аллоцировать часы день за днём
     - Обработать зависимости (заблокированная стори ждёт завершения блокера)
5. Если план > 130 рабочих дней → быстрый режим (ceil(hours/capacity))

### Competency Factor
- Score 1 → ×1.6 (60% медленнее), Score 3 → ×1.0, Score 5 → ×0.7 (30% быстрее)

### Предупреждения
- NO_ESTIMATE: нет оценок подзадач
- NO_CAPACITY: нет исполнителя для роли
- FLAGGED: работа приостановлена
- CIRCULAR_DEPENDENCY: циклическая зависимость

## РЕТРОСПЕКТИВНЫЙ ТАЙМЛАЙН

Показывает фактическое прохождение фаз в прошлом.

- Гибридный merge: ретро (сплошные бары) + прогноз (полосатые бары)

## FAQ: НАВИГАЦИЯ

### Где посмотреть текущие задачи?
Board (/) — главная страница. Выбери команду в фильтре.

### Где метрики команды?
Metrics (/metrics) — выбери команду и период.

### Где прогноз завершения?
Timeline (/timeline) — полосатые бары = прогноз. Наведи для деталей.

### Где прогресс проектов?
Projects (/projects) — прогресс-бары и RICE. Project Timeline (/project-timeline) — Гант.

### Где настроить маппинг Jira?
Workflow Config (/workflow) — только ADMIN. 4 вкладки.

### Как добавить участника?
Teams → выбери команду → «+ Add Member». Укажи роль, грейд, часы.

### Как назначить роль?
Settings (/settings) → найди пользователя → измени роль. Только ADMIN.

### Как запустить синхронизацию?
Settings → «Sync Now». Или автоматически каждые 5 минут.

### Где проблемы с данными?
Data Quality (/data-quality) — все нарушения с ссылками на Jira.

### Как настроить SLA?
Settings → вкладка Bug SLA → установи часы для каждого приоритета.

### Как добавить отпуск?
Teams → команда → секция «Отпуска» → «+ Добавить».

### Как изменить порядок эпиков?
Board (/) → выбери одну команду → перетащи эпик. Порядок сохранится.

### Как оценить стори?
Poker (/poker) → создай сессию → поделись кодом → голосуйте → раскрывайте.

### Где компетенции команды?
Teams → команда → «Competency Matrix». Матрица + bus-factor алерты.

### Как зарегистрировать компанию?
/register → название → slug → первый пользователь = ADMIN.

### Где RICE-скор?
Projects (/projects) — бэдж RICE. Кликни на проект для формы оценки.

### Почему DSR красный?
DSR > 1.5 = перерасход 50%+. Причины: неточные оценки, блокировки, высокий WIP.

### Что делать если прогноз опаздывает?
Timeline → полосатый бар выходит за красную линию. Решения: пересмотр scope, доп. люди, переприоритизация.

### Где загруженность ролей?
Metrics (/metrics) → секция Role Load — SA/DEV/QA.

### Как работает auto-detect типов?
Workflow Config → при синке анализирует названия: «bug/баг» → BUG, «epic/эпик» → EPIC. Неизвестные → NEW badge.

### Что означает confidence на прогнозе?
HIGH (зелёный) = все оценки есть. MEDIUM (жёлтый) = частично. LOW (красный) = нет оценок.
