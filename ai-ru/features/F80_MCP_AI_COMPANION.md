## 1. Цель и идея

Дать тимлиду **ИИ-компаньона**, который анализирует задачи, время, capacity и сроки команды и подсказывает, что брать в работу, что горит и что застряло. Компаньон должен быть «человечным» — общение в чате с компьютера и с телефона.

**Ключевое архитектурное решение:** не строить собственный LLM-стек и UI чата. Вместо этого выставить **remote MCP-сервер** над Lead Board, к которому подключается **приложение Claude** (мобильное + веб + десктоп). Claude выступает агентом и LLM; наш сервер даёт ему «руки» к данным доски через инструменты (tools).

**Роль ИИ — аналитик-компаньон, а не автономный движок.** Детерминированные расчёты (AutoScore F19, Matrix F78, Planning F21/F55, Status Age F79, RICE F36) остаются источником цифр. ИИ их интерпретирует, объясняет человеческим языком и подсвечивает то, что формулы структурно не ловят. Это минимизирует риск галлюцинаций: цифры детерминированы, LLM только формулирует.

### Подтверждение ценности на реальных данных

Перед проектированием проведён «dogfood»-анализ на схеме `tenant_test2` (589 задач). Подтверждено, что движок инсайтов даёт реальную ценность и что приоритеты тимлида — это:

1. **Готовность к планированию** — 101 активная задача без квартальной метки, Q3 пуст.
2. **Параллельная загрузка / простой** — worklog за 90 дней списан на одного человека (1332 ч, что физически невозможно → метрики по людям недостоверны, нужно чинить источник).
3. **Качество данных как блокер решений** — 81 задача без оценки, 49 без исполнителя, 1 дедлайн из 589.
4. **Поток** — бутылочное горло на стадии аналитики (32 истории в `Waiting SA`); эпик LB-302 под угрозой (дедлайн прошёл, 2/5 историй не начаты).

Эти 4 «линзы» в этом порядке приоритета формируют движок инсайтов (см. §5).

## 3. Архитектура

```
Телефон / ПК: приложение Claude        ← агент + LLM (не наш)
        │  HTTPS / Streamable HTTP + OAuth 2.1 Bearer
        ▼
   nginx (79.174.94.70) ──► backend (Spring Boot 3.2.5, Java 21)
        │
   ┌────┴─────────────────────────────────────────┐
   │  MCP Resource Server (/mcp)                    │
   │   • MCP Java SDK, Streamable HTTP транспорт    │
   │   • валидация Bearer → user + tenant + role    │
   │   ├── MCP Tool Adapter ──► ChatToolRegistry/Executor (F52, read-only)
   │   └── team_readiness_briefing ──► InsightEngine (4 линзы)
   │                                                 │
   │  OAuth 2.1 Authorization Server (/oauth/mcp/*)  │
   │   • Spring Authorization Server 1.2.x           │
   │   • authorize / token / register(DCR) / revoke  │
   │   • аутентификация юзера через существующий      │
   │     Atlassian 3LO + сессию Lead Board           │
   │   • реестр выданных токенов (mcp_tokens)         │
   └─────────────────────────────────────────────────┘
                    │
              Postgres (schema-per-tenant)
```

**Транспорт:** Streamable HTTP (SSE deprecated, sunset mid-2026). Endpoint `/mcp`.

**MCP-сервер:** «голый» MCP Java SDK (`io.modelcontextprotocol.sdk`) со servlet-транспортом, **без Spring AI** — чтобы не делать апгрейд Spring Boot ради MCP-части. MCP-сервер — изолированный модуль.

## 4. Безопасность (детально)

### 4.1. Авторизация через Claude (OAuth 2.1)

Backend **впервые становится OAuth Authorization Server** (сейчас он только клиент к Atlassian). Новых паролей не заводит — переиспользует вход через Atlassian как способ установить личность.

Поток (Authorization Code + PKCE S256 + Dynamic Client Registration):

1. В claude.ai добавляется коннектор → URL `https://board.<домен>/mcp`.
2. Claude дёргает `/mcp` без токена → **401** + `WWW-Authenticate: Bearer ... resource_metadata="<url>"`.
3. Claude читает `/.well-known/oauth-protected-resource` и `/.well-known/oauth-authorization-server` → узнаёт endpoints.
4. **DCR**: Claude регистрируется как клиент → `POST /oauth/mcp/register` → `client_id`.
5. Claude открывает в браузере `/oauth/mcp/authorize?...&code_challenge=...`.
6. Authorize-endpoint проверяет активную сессию Lead Board:
   - нет → редирект на привычный вход через Atlassian (3LO);
   - есть → экран согласия («Claude получит доступ от вашего имени, только чтение, команды: …»).
7. Согласие → редирект в Claude с одноразовым `authorization code`.
8. Claude меняет code на **access token (короткий TTL) + refresh token** на `/oauth/mcp/token` с PKCE verifier.
9. Каждый запрос к `/mcp` идёт с `Authorization: Bearer <токен>`.
10. Resource server валидирует токен → `user_id`, `tenant`, `role`, `scopes` → ставит `TenantContext` + `SecurityContext`.

**Важно:** claude.ai/мобильный поддерживают **только OAuth** (статический токен/заголовки нельзя). Поэтому OAuth обязателен для цели «с телефона». Bearer-токен — только для локальной отладки через Claude Code CLI (`--header`) / MCP Inspector.

### 4.2. Разграничение прав (RBAC) — наследуется из F27

- Токен несёт личность → per-tenant роль (`tenant_users.app_role`) + команды пользователя (`team_members` по `jira_account_id`, `active=true`).
- Каждый инструмент исполняется с правами пользователя через `AuthorizationService`:
  - **ADMIN / PROJECT_MANAGER** → все команды tenant;
  - **TEAM_LEAD / MEMBER** → только свои; `teamId` принудительно фильтруется. Чужая команда → пусто/403, не утечка.
- **OAuth scopes — второй контур:**
  - `mcp:read` — все инструменты-читалки (фаза 1);
  - `mcp:write` (фаза 2) — изменяющие действия, отдельное согласие.
- MCP не обходит RBAC: инструменты вызывают те же проверки, что REST-контроллеры.
- **Tenant-изоляция железная:** tenant берётся из токена; заголовок `X-Tenant-Slug` для MCP игнорируется.

### 4.3. Отзыв доступа и увольнение

Принцип: **отзыв работает на нашем сервере, не полагаясь на Claude** (Anthropic не гарантирует вызов revocation endpoint при удалении коннектора).

1. **Короткое окно компрометации:** access token TTL 15 мин; refresh token — дни/недели, проверяется по реестру при каждом обновлении.
2. **Реестр токенов** (новая таблица `mcp_tokens`): `user_id, tenant_id, client_id, scopes, refresh_token_hash, issued_at, expires_at, revoked_at, last_used_at`. Refresh-токены — только хэш.
3. **Увольнение (каскад):** деактивация `team_members.active=false` / удаление из `tenant_users` → пометить все `mcp_tokens` пользователя `revoked_at=now()`. При каждом запросе resource server проверяет: токен не отозван И пользователь активен в tenant И (для team-scoped) есть активные команды → иначе **401**. Уволенный теряет доступ ≤15 мин (или мгновенно на refresh/team-проверке).
4. **Явный отзыв админом:** экран «Активные подключения Claude» (из `mcp_tokens` + `last_used_at`) → «Отозвать» → `revoked_at=now()`. Плюс **RFC 7009 `/oauth/mcp/revoke`** для корректного отзыва Claude.
5. **Аудит:** каждый вызов инструмента → лог `user_id, tenant, tool, params-hash, timestamp` (без секретов/PII в значениях); обновление `last_used_at`.
6. **Сетевые контуры (F46):** HTTPS обязателен; rate limit группа `mcp`; валидация `aud` токена; шифрование refresh-токенов через `EncryptionService`.

## 5. Движок инсайтов: `team_readiness_briefing`

Новый MCP-инструмент. Вход: `teamId` (опц.; без него — по всем доступным пользователю командам). Выход: структурированный JSON (Claude озвучивает человечно). Цифры детерминированы, со ссылками на задачи.

Четыре линзы в порядке приоритета:

1. **planning_readiness** — доля задач без квартальной метки; покрытие оценок (`rough_estimates`); наличие очередности (`auto_score` / `manual_order`). Статус: red/yellow/green + конкретные ключи.
2. **load_balance** — параллельная загрузка ролей и людей (capacity F55 vs назначенный объём; worklog-активность). **Самокритика по достоверности:** если 100% worklog на одном человеке или цифры нереальны (> физического максимума) → флаг «данные недостоверны, доверять загрузке нельзя».
3. **data_quality_blockers** — задачи без оценки / без исполнителя / без дедлайна как блокеры качественных решений (не абстрактный список, а «вот почему совет ограничен»).
4. **flow** — бутылочное горло (самая длинная активная стадия), эпики под угрозой (F79 stuck + дедлайн прошёл + незапущенные истории).

**Анти-шум:** схлопывать кластеры с одинаковым `daysInStatus` (артефакт данных), приглушать подозрительно «ровные» цифры.

## 6. Модель данных (новые таблицы)

Public schema (миграции `V{N}__`), т.к. OAuth-клиенты и токены — кросс-tenant:

- **`oauth2_registered_client`** — стандартная таблица Spring Authorization Server (DCR-клиенты Claude).
- **`oauth2_authorization`** — стандартная таблица SAS (authorization codes, токены).
- **`mcp_tokens`** — наш реестр для отзыва/аудита (см. §4.3). Может быть представлением/надстройкой над `oauth2_authorization` либо отдельной таблицей — решить при реализации.

Связь tenant: `client_id` ↔ `user_id` ↔ `tenant_id` фиксируется в момент авторизации (шаг 6–8).

## 7. Поэтапный план реализации

Каждый этап — проверяемый результат. Цель: подключение с ПК → затем с телефона.

- **Этап 0. Каркас.** F80, bump 0.80.0. Зафиксировать решения (SDK, SAS, scopes).
- **Этап 1. MCP «hello» (локально).** MCP Java SDK, `/mcp` с одним tool `ping`. Результат: сервер отвечает по MCP-протоколу.
- **Этап 2. Мост к F52 (локально).** Адаптер `ChatToolRegistry` → MCP tools list; вызов → `ChatToolExecutor`, read-only. Результат: 14 инструментов видны через MCP (проверка через MCP Inspector).
- **Этап 3. Инструмент-аналитик.** `team_readiness_briefing` (4 линзы, §5). Результат: Claude выдаёт «утренний брифинг».
- **Этап 4. Отладочная аутентификация (локально).** Bearer-токен из `.env` → user/tenant. Результат: подключение с ПК через **Claude Code CLI `--header`** — первая локальная победа. 🎯
- **Этап 5. OAuth 2.1 Authorization Server.** Spring Authorization Server 1.2.x: authorize/token/register/revoke + well-known metadata + интеграция с Atlassian-входом + реестр `mcp_tokens` + отзыв (§4.3). Результат: полноценный OAuth flow.
- **Этап 6. Деплой.** nginx proxy `/mcp` и `/oauth/mcp/*`, HTTPS (Let's Encrypt) на 79.174.94.70. Результат: публичный защищённый endpoint.
- **Этап 7. Подключение с телефона.** claude.ai (веб) → Connectors → добавить custom connector по URL → синхронизация на телефон. Результат: **управление доской с телефона** — финальная цель. 🎯🎯

## 8. Вне scope (YAGNI)

- Свой LLM/чат-UI — заменён приложением Claude.
- Предиктивные ML-модели — никто из конкурентов их не использует; детерминированные сигналы + LLM-формулировка достаточны.
- Per-person overload-инсайты — пока недостоверны данные worklog; сначала чинить источник (отдельная задача).
- Автономные действия ИИ — только ассистивный режим с подтверждением человеком.

## 9. Тестирование

- **Безопасность (критично):** OAuth flow (PKCE, DCR, code exchange); отзыв при деактивации участника → 401 ≤15 мин; tenant-изоляция (токен tenant A не видит tenant B); RBAC (TEAM_LEAD не видит чужие команды через MCP); валидация `aud`; rate limit `/mcp`.
- **Инструменты:** адаптер отдаёт корректный MCP tools list; `team_readiness_briefing` на сидовых данных даёт ожидаемые 4 линзы; анти-шум схлопывает одинаковые кластеры.
- **Регресс:** существующий F52 чат не сломан (общая логика инструментов).
- JUnit5, `./gradlew test` зелёный перед коммитом.
