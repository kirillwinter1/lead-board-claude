## Fix 1: Priority Tooltip Debounce (PriorityCell.tsx)

### What was fixed
Tooltip on priority score was triggering an API call (`/api/board/score-breakdown`) immediately on mouse enter, causing slow loading on production with many issues.

### Changes
- Added `debounceRef` for 300ms delay before API call
- Tooltip UI shows instantly on hover (no delay for visual feedback)
- API call fires only after 300ms of hovering
- Mouse leave cancels pending debounce timer and aborts in-flight request

### QA Result: PASS
- Tooltip appears instantly on hover
- API call is debounced (verified: no request fires on quick mouse-over)
- AbortController properly cancels on mouse leave
- No memory leaks (cleanup on unmount not needed â€” component stays mounted)

## Fix 2: Rough Estimate Optimistic Update (useBoardData.ts)

### What was fixed
Entering a rough estimate caused full board reload (`GET /api/board`), making the entire table "blink".

### Changes
1. Replaced `fetchBoard()` call with optimistic state patch
2. Recursive `patchNode` traverses nested tree (Project > Epic)
3. Patches both `node.roughEstimates` AND `node.roleProgress[role].roughEstimateDays`

### Bug Found During QA
**Severity: High**
**Description:** Initial optimistic update only patched `node.roughEstimates`, but `RoleChips.tsx` reads from `node.roleProgress[role].roughEstimateDays` when `roleProgress` exists. UI did not visually update after save.

**Root cause:** `RoleChips.tsx` line 39-44 prioritizes `roleProgress` over `roughEstimates`:
```typescript
const metrics: RoleMetrics = roleProgress?.[code] ?? {
  estimateSeconds: 0, loggedSeconds: 0, progress: 0,
  roughEstimateDays: node.roughEstimates?.[code] ?? null
}
```

**Fix:** Added `roleProgress` patching in `handleRoughEstimateUpdate` (commit f1c2c93).

### QA Result after fix: PASS
- No GET /api/board request after saving rough estimate (verified via network)
- Only 1 PATCH request fires
- UI updates instantly without blinking
- Recursive tree patching works for nested epics under projects
- Type safety maintained (TypeScript compiles clean)

## Recommendations
- Consider adding a lightweight frontend test for `handleRoughEstimateUpdate` to verify both `roughEstimates` and `roleProgress` are patched
- The `getScoreBreakdown` API could benefit from server-side caching if production latency remains an issue
