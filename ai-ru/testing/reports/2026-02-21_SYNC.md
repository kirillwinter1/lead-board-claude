## Bugs Found

### Critical

#### SYNC-001: Зависший sync_in_progress без механизма восстановления
**Severity:** Critical
**Файл:** `SyncService.java:168-179`, `SyncService.java:303-308`

**Описание:** Флаг `sync_in_progress=true` в БД зависает навсегда, если приложение перезапускается или sync-поток падает. Нет timeout/TTL, нет auto-recovery при старте приложения.

**Воспроизведение:**
1. `GET /api/sync/status` → `syncInProgress: true, lastSyncStartedAt: 2026-02-19T15:09:41` (2 дня назад!)
2. `POST /api/sync/trigger` → `"error": "Sync already in progress"`
3. Все scheduled syncs тоже пропускаются (проверка на строке 171)

**БД:**
```sql
SELECT sync_in_progress, last_sync_started_at FROM jira_sync_state;
-- sync_in_progress=true, last_sync_started_at=2026-02-19 18:09:41 (2 дня назад)
```

**Impact:** Система полностью заблокирована — ни ручная, ни автоматическая синхронизация не работает.

**Ожидаемое поведение:**
- При старте приложения: сбросить `sync_in_progress` если `lastSyncStartedAt` > 30 мин назад
- Или: добавить timeout — если sync длится > N минут, считать зависшим

#### SYNC-002: Reconciliation может удалить ВСЕ задачи при ошибке Jira API
**Severity:** Critical
**Файл:** `SyncService.java:614-654`

**Описание:** Если Jira API возвращает пустой ответ (200 с пустым списком), `reconcileDeletedIssues()` считает что в Jira 0 задач → удаляет ВСЕ локальные задачи.

**Сценарий:**
1. Jira API вернул пустой ответ (токен истёк, сетевой сбой, проект переименован)
2. `jiraKeys` = пустой Set
3. ВСЕ `dbKeys` попадают в `orphanedKeys`
4. `issueRepository.deleteByIssueKeyIn(orphanedKeys)` → **массовое удаление**

**Код (строки 627-645):**
```java
if (issues == null || issues.isEmpty()) break; // выходим с пустым jiraKeys
...
List<String> orphanedKeys = dbKeys.stream()
    .filter(key -> !jiraKeys.contains(key)) // ВСЕ ключи "orphaned"
    .toList();
if (!orphanedKeys.isEmpty()) {
    issueRepository.deleteByIssueKeyIn(orphanedKeys); // УДАЛЕНИЕ ВСЕХ
}
```

**Impact:** Потеря всех данных в локальной БД.

**Ожидаемое поведение:** Sanity check: не удалять если `jiraKeys` пуст или `orphanedKeys.size() > N% от dbKeys.size()`.

### High

#### SYNC-003: @Transactional не работает при self-invocation через Thread
**Severity:** High
**Файл:** `SyncService.java:132-136`

**Описание:** `triggerSync()` создаёт `new Thread(() -> { syncProject(...); })` — это прямой вызов `this.syncProject()`, который обходит Spring AOP proxy. Аннотация `@Transactional` на `syncProject()` (строка 167) и `reconcileDeletedIssues()` (строка 614) **не действует**.

**Аналогично:** `scheduledSync()` (строка 106) вызывает `this.syncProject()` — тоже обход proxy.

**Impact:**
- Sync не атомарен — при crash часть задач сохранена, часть нет
- `sync_in_progress` может остаться true при частичном сбое (усугубляет SYNC-001)

**Fix:** Inject self-reference `@Lazy SyncService self` или использовать `TransactionTemplate`.

#### SYNC-004: jira_updated_at не заполняется при sync
**Severity:** High
**Файл:** `SyncService.java:337-528`, `JiraIssue.java:51-73`

**Описание:** Поле `jira_updated_at` в `JiraIssueEntity` (строка 118-119) **никогда не записывается** в `saveOrUpdateIssue()`. Jira API отдаёт поле `updated`, но `JiraFields` его не парсит (нет такого поля в DTO). Результат:

```sql
SELECT MIN(jira_updated_at), MAX(jira_updated_at) FROM jira_issues;
-- NULL, NULL (все 246 записей)
```

**Impact:**
1. Changelog import `months` фильтр бесполезен — использует `updated_at` (local), которое обновляется при каждом sync → всегда возвращает ВСЕ задачи:
   ```
   GET /api/sync/import-changelogs/count?months=1 → issueCount=246, totalIssues=246
   GET /api/sync/import-changelogs/count?months=3 → issueCount=246, totalIssues=246
   ```
2. Нет данных о реальном времени обновления в Jira для аналитики

### Medium

#### SYNC-005: Raw Thread без error handling и naming
**Severity:** Medium
**Файл:** `SyncService.java:132-136`

**Описание:** `new Thread(() -> { ... }).start()` — не managed executor. Uncaught exceptions в sync thread теряются (не логируются). Нет имени потока — сложно дебажить.

**Ожидаемое:** `@Async` с настроенным `TaskExecutor` или `CompletableFuture.runAsync()`.

---

#### SYNC-006: Нет защиты от конкурентного changelog import
**Severity:** Medium
**Файл:** `ChangelogImportService.java:59-67`

**Описание:** Несколько `POST /api/sync/import-changelogs` запускают параллельные async импорты. Нет флага `importInProgress`. Может привести к дублированным записям в `status_changelog`.

**Воспроизведение:**
```
POST /api/sync/import-changelogs → 200 "started"
POST /api/sync/import-changelogs → 200 "started" (сразу)
```

---

#### SYNC-007: Отрицательные months принимаются без валидации
**Severity:** Medium
**Файл:** `SyncService.java:656-675`, `SyncController.java:33-35`

**Описание:** `months=-1` возвращает 200 с данными. JQL генерируется как `updated >= --30d` — может вызвать неожиданное поведение Jira API.

**Воспроизведение:**
```
GET /api/sync/issue-count?months=-1 → {"months":-1,"total":246} (200 OK)
```

**Ожидаемое:** Валидация `months >= 0`, 400 Bad Request при отрицательных значениях.

---

#### SYNC-008: Транзиентная ошибка 500 при issue-count
**Severity:** Medium
**Файл:** `SyncController.java:31-36`

**Описание:** Первый запрос `GET /api/sync/issue-count?months=6` вернул 500 Internal Server Error. Повторный запрос вернул 200 OK. Вероятная причина — timeout Jira API или rate limiting.

**Воспроизведение:** Нестабильно (транзиентная ошибка).

**Ожидаемое:** Retry логика или информативное сообщение об ошибке (а не generic 500).
